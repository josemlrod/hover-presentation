The solution was an interactive four-step flow that let visitors experience Transmit's technology firsthand.
Step 1: Upload content or use provided defaults.
Step 2: Create their ad unit by uploading whatever assets are applicable to their chosen ad format or use provided defaults.
Step 3: Select exactly where in the content to insert the ad.
Step 4: Enter their email to trigger the encoding process which ultimately sends the user the created encode via email.

Technical Architecture: Frontend

The architecture was designed for both user experience and scalability.
On the frontend, we used the nested routes approach because it allowed us to do the following:
- Keep the flow simple and easy to understand
- Keep the validation of each step separated
- Keep the layout of the flow consistent
The parent layout route held the context for all the user choices on each step. This enabled us to also grant/refuse access to next steps, and prompt the user to confirm before navigating away since their choices could be lost.
The child routes (steps) were responsible for rendering the form/content, enforcing requirements from their previous/next step to prevent user errors, and validating user input.

Technical Architecture: Backend

The backend was designed to be scalable and reliable.
Speak to `createYourOwnAdHandler`:
- It's the main handler for the create your own ad flow.
- It's responsible for:
- Validating the request, and then sending a message to the message queue.
- Creating a `jobProcessing` record in the database.
- Sending a response to the client, including a `jobId` to track the encode.
Speak to `processCreateYourOwnAd`:
- It's the worker that processes the messages from the queue.
- It's responsible for:
- Downloading the assets from the file storage service.
- Creating the encode using ffmpeg.
- Uploading the result to the file storage service.
- Sending an email to the user with the URL to the result.
- Updating the `jobProcessing` record in the database.
We're storing the `jobProcessing` record in a Non-relational DB to track the encoding progress.
Something we didn't metion before, but should now is that the last step in the flow, after submitting the encode it starts polling to get the status of the job until the job is completed. That allows us to provide the user with some feedback on the status of the job.

Challenge #1: File Upload Performance

Problem

  • Users uploading large video files could experience long wait times, and potentially drop-off from the flow due to network latency.
The first major challenge was keeping users engaged during file uploads. Large video files could take minutes to upload, and we were worried about drop-offs. Our solution was multi-pronged: we set reasonable file size limits, provided high-quality default videos so users could experience the flow immediately, and implemented direct S3 uploads to avoid bottlenecking through our servers. We also added clear progress indicators. The key insight was that users who just wanted to test the technology could use defaults, while serious prospects would be willing to wait for their own content.

Solution

  • Implemented file size limits for reasonable upload times
  • Provided "Use default video" options for immediate testing
  • Direct S3 uploads bypassing our servers
  • Progress indicators and upload status feedback

Outcome

  • Result: Users who started the flow completed it - no significant drop-offs during upload steps.

Challenge #2: Scalable Video Processing

Problem

  • Video encoding takes significant time/resources. Risk of losing submissions during high traffic.
The second challenge was handling the video processing reliably at scale. Video encoding with FFmpeg is CPU-intensive and time-consuming - downloading files, processing video, uploading results, and sending emails could take several minutes per job. We couldn't do this synchronously without terrible user experience. The solution was a robust async architecture: when users submitted, we immediately queued the job and gave them confirmation. Lambda workers would pick up jobs as capacity allowed, process them completely, and send results via email. We monitored the ratio of submissions to completions to ensure 100% reliability.

Solution

  • Async message queue (AWS SQS)
  • Lambda workers for auto-scaling
  • Pipeline: download → process → upload → email
  • Monitor submission/completion parity

Outcome

  • Result: 100% reliability - every submission completed successfully.

Security & Performance

Security Measures

  • Rate Limiting: 10 requests/minute per IP
  • File Validation: Type and size restrictions
Security and performance were critical considerations. We implemented rate limiting to prevent abuse of the encoding endpoint - 10 requests per minute per IP address was sufficient for legitimate use while preventing spam. File validation ensured only appropriate video formats and sizes were processed.

Performance Monitoring

  • Success Rate: Frontend submissions vs. completed encodes
  • Processing Time: Average encode completion time
  • User Analytics: FullStory for behavior insights
  • Error Tracking: Failed uploads and processing jobs

Lessons Learned & Future Improvements

Key Learnings

  • Async Architecture: Essential for video processing workloads
Several key lessons emerged from this project. First, providing default options was crucial - many users just wanted to test the functionality quickly rather than upload their own content.
